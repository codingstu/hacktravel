# AI 可靠性与记忆机制

关联文档：
- 蓝图：[`implementation-blueprint.md`](plans/implementation-blueprint.md)
- 协作规范：[`工程协作与Agent-Skills规范.md`](docs/2026-03-09/工程协作与Agent-Skills规范.md)
- 执行清单：[`实施执行清单.md`](docs/2026-03-09/实施执行清单.md)

## 1. 目标

确保后续所有 Agent 与大模型调用满足以下底线：

- 尽量降低幻觉
- 不因会话切换丢失关键约束
- 不因上下文过长遗忘系统规则、蓝图和当前任务状态
- 输出必须可追溯、可校验、可回滚

## 2. 核心原则

### 2.1 事实源优先

所有模型输出必须优先服从“可验证事实源”，禁止凭空补全。

事实源优先级固定为：

1. [`implementation-blueprint.md`](plans/implementation-blueprint.md)
2. 当日 `docs/YYYY-MM-DD/*.md`
3. `agents/` 角色文件
4. `skills/` 技能文件
5. `prompts/` 模板文件
6. 当前代码与配置
7. 用户本轮明确指令

如果事实源之间冲突，必须以最新的蓝图或用户最新明确指令为准，并在文档中登记。

### 2.2 短上下文只做执行，长记忆靠文件

不能把“记忆”依赖在单次对话上下文里，必须外置到仓库文件：

- 蓝图放在 [`implementation-blueprint.md`](plans/implementation-blueprint.md)
- 当前规范放在 [`工程协作与Agent-Skills规范.md`](docs/2026-03-09/工程协作与Agent-Skills规范.md)
- 每个功能交付放在 `docs/YYYY-MM-DD/功能点.md`
- Agent 行为边界放在 `agents/*.md`
- 技能流程放在 `skills/*.md`
- Prompt 约束放在 `prompts/*.md`

结论：即使会话丢失，只要重新读取这些文件，就能恢复项目状态。

### 2.3 输出必须结构化

凡是关键路径输出，都必须结构化，禁止自由发挥长文本：

- API 输出必须走严格 Schema
- 任务卡必须按模板输出
- 功能交付必须包含变更文件、验收结果、回滚方式、commit hash
- 故障修复必须包含根因分析与复现步骤

## 3. 防幻觉机制

### 3.1 强约束输入

每次调用模型前，必须显式注入：

- 当前功能点范围
- 非范围内容
- 事实源文件列表
- 输出格式
- 禁止事项

### 3.2 强约束输出

模型输出后必须经过两层校验：

1. 结构校验：字段是否完整、格式是否正确
2. 业务校验：是否违反蓝图、是否越界、是否出现未定义字段或假设

### 3.3 未知即未知

如果模型缺少事实依据，必须返回以下之一：

- “当前事实源中未定义”
- “需要补充配置/文件后才能确认”
- “按现有蓝图无法安全推断”

禁止伪造：

- 不存在的接口
- 不存在的环境变量
- 不存在的第三方配置
- 不存在的提交记录

### 3.4 工具结果优先于推测

凡是能通过工具确认的事实，必须用工具确认，不能靠猜：

- 文件存在性用目录/文件工具确认
- Git 状态用命令确认
- 配置内容用读文件确认
- 代码结构用读文件确认

## 4. 防失忆机制

### 4.1 每轮任务都要先恢复上下文

开始新功能点前，固定执行恢复动作：

1. 读取 [`implementation-blueprint.md`](plans/implementation-blueprint.md)
2. 读取最近一次功能文档 `docs/YYYY-MM-DD/功能点.md`
3. 读取对应 Agent 文件
4. 读取对应 Skill 文件
5. 再开始实现

### 4.2 长任务必须做阶段性摘要

当功能点较大时，必须把阶段状态写入 docs，而不是只存在聊天记录里。

建议每完成一个阶段，就写入：

- 当前完成了什么
- 下一步做什么
- 关键约束是什么
- 存在什么风险

### 4.3 当前状态以仓库为准

项目状态判断优先基于仓库文件，而不是基于模型“记得什么”。

也就是说：
- 是否完成，看 docs 和记 Git 记录
- 是否有规范，看 `agents/`、`skills/`、`prompts/`
- 是否允许某行为，看蓝图和规范文件

## 5. 实施层面的落地要求

### 5.1 Prompt 层

所有实现 Prompt 必须包含：

- 功能边界
- 非范围
- 事实源引用
- 输出模板
- 文档更新要求
- Git 要求

### 5.2 API 层

后端模型输出必须：

- 使用严格 Schema 校验
- 增加版本号
- 增加错误码
- 增加 provider / model / cache_hit 元数据

### 5.3 文档层

每次功能点完成后，必须新增文档以持久化记忆。

文档至少记录：

- 做了什么
- 改了哪些文件
- 约束是否变化
- 有无新增风险
- commit hash

## 6. 最小执行 SOP

每次使用大模型执行任务时，按这个顺序：

1. 先读蓝图
2. 再读最近 docs
3. 再读对应 Agent / Skill / Prompt
4. 明确本次功能范围与非范围
5. 执行修改
6. 校验输出是否越界
7. 更新 docs
8. Git 提交并 push

## 7. 验收标准

满足以下条件，才算“不会因上下文变化而失忆”：

- 关键约束都已经落入仓库文件
- 任意新会话只读仓库文件即可恢复项目状态
- 模型输出必须引用事实源，不允许自由脑补
- 任何功能点都能在 docs 中找到变更记录与提交记录

## 8. 当前结论

要避免幻觉和失忆，不能只靠模型能力本身，必须采用“文件化记忆 + 结构化输出 + 工具校验 + 文档回填”的组合方案。

这也是当前仓库规范设计的核心原因。
