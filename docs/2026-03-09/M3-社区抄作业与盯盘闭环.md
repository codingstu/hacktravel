## 1. 功能目标

交付 M3 阶段的真实可用闭环，覆盖：
- Tab2「抄作业」从前端预置数据升级为后端社区路线接口驱动
- Tab2 支持路线详情加载、「我也要抄」计数递增、地点详情弹窗
- Tab3「盯盘」接入真实邮箱收集、订阅人数统计、价格提醒创建与列表查询
- Tab1 区域筛选、区域元数据读取、GPS/时区推断与重新规划链路完善
- 补齐对应的后端模型、路由、服务与测试，形成 API + 前端页面 + Redis 持久化闭环

## 3. 关键实现

### 3.1 Tab2 社区路线真实后端化

- 使用 `preset_routes.py` 作为真实路线事实源，而不是前端静态 mock
- 在 `community_service.py` 中把预置路线转换为社区卡片与详情模型
- copy_count 通过 Redis key `hkt:community:copy:{route_id}` 持久化
- 「我也要抄」调用后端 `copy` 接口后原地更新计数，并可直接打开 Google Maps deeplink
- 展开路线详情时按需加载完整 legs，减小列表首屏负担

### 3.2 Tab3 盯盘闭环

- `POST /v1/leads`：邮箱按 SHA-256 做去重，重复提交返回成功且标记 `is_duplicate=true`
- `GET /v1/leads/stats`：返回实时订阅人数与展示文案
- `POST /v1/watchlist/alerts`：创建价格提醒，单邮箱最多 10 条
- `GET /v1/watchlist/alerts?email=...`：按邮箱拉取提醒列表
- 前端 watchlist 页面支持创建提醒、查看提醒、显示订阅人数和错误反馈

### 3.3 Tab1 区域化体验增强

- 通过 `fetchRegionMetadata()` 从后端读取大洲、子区域、热点目的地元数据
- 当后端不可用时，前端回退到 `REGION_METADATA_FALLBACK` 与 `FEATURED_SUB_REGIONS_FALLBACK`
- 首次进入时优先请求 GPS，经纬度推断失败再回退时区推断
- 重新规划通过 `skip_preset=true` 强制跳过预置路线，避免重复命中秒出结果
- 地点详情弹窗可从 itinerary 与 community 两个 Tab 复用

### 3.4 测试与工程配套

- 后端新增 community / leads / watchlist / places 测试文件，覆盖模型校验与关键服务逻辑
- 前端新增 `region` 助手测试，覆盖时区推断、坐标推断、子区域热点回退逻辑
- 新增 Jest 配置，使区域工具与前端服务具备可持续回归基础
- Expo dev server 已验证可启动，Docker 后端已验证重新加载最新 `backend/.env`

## 5. 风险与回滚

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| Redis 不可用 | 社区 copy_count、邮箱订阅、价格提醒失效或回退 | 服务层对 Redis 异常做降级，前端保留兜底提示 |
| 地点详情第三方来源波动 | 地点弹窗信息可能为空 | 弹窗保留基础地点名与地图链接，不阻塞主流程 |
| GPS 授权被拒绝 | 默认区域推断不够精准 | 已回退到时区推断与预置元数据 |
| 社区路线仍基于预置数据 | 热门度为 seed + copy 累积，并非完整 UGC | 后续可接入真实投稿与审核链路 |

回滚方式：
- 前端层面可将 Tab2、Tab3 退回到预置数据与静态占位文案
- 后端层面可移除 `community`、`leads`、`watchlist`、`places` 路由挂载，仅保留 M1/M2 主链路
- 若区域元数据接口异常，可仅保留前端 fallback 常量，不影响基础 itinerary 生成
