## 1. 功能目标

在用户点击"开始规划"时，优先匹配内置预置路线（0ms 响应），仅在无匹配时回退至 LLM 生成（10-60s）。
核心目标：消除 80%+ 热门目的地查询的长等待，降低用户流失率。

## 3. 关键实现

### 3.1 预置路线匹配算法

```
preset lookup (0ms) → idempotency → cache (1ms) → LLM (10-60s)
```

匹配逻辑（`find_preset()`）：
1. **目的地模糊匹配**：支持中英文别名（如 `okinawa` → `冲绳`、`saigon` → `胡志明市`），通过 `_DEST_ALIASES` + `_normalize_dest()` 实现
2. **区域优先过滤**：若请求传入 `continent` / `sub_region`，先在对应区域候选集内匹配
3. **区域降级回退**：若指定区域无候选，则自动回退到全局同目的地候选，避免误伤可命中路线
4. **时长容忍**：±50%（如 48H 预置可匹配 24H-72H 查询）
5. **预算容忍**：±60%
6. **标签加分**：tag 交集/并集比率 × 权重，非必要条件
7. **评分阈值**：score < 5.0 视为不匹配

### 3.2 预置路线覆盖

当前预置路线已升级为**全球区域化组织**，按 6 大洲与重点子区域分层：

- **Asia / EastAsia**：冲绳、东京、大阪、福冈、札幌、首尔、台北等
- **Asia / SoutheastAsia**：曼谷、清迈、新加坡、普吉岛、槟城、巴厘岛、胡志明市等
- **Europe / UK**：伦敦、爱丁堡、曼彻斯特、利物浦
- **Europe / ContinentalEurope**：巴黎、罗马、巴塞罗那、阿姆斯特丹
- **SouthAmerica / LatinAmerica**：里约热内卢、圣保罗、布宜诺斯艾利斯、利马、圣地亚哥等
- **Africa / NorthAfrica / SubSaharanAfrica**：开罗、马拉喀什、开普敦、内罗毕、桑给巴尔等

重点覆盖密度要求：

| 区域 | 目标 |
|------|------|
| 东亚 | ≥ 6 条路线，覆盖 24H / 48H / 72H |
| 东南亚 | ≥ 6 条路线，覆盖 24H / 48H / 72H |
| 英国 | ≥ 4 条路线，覆盖 24H / 48H / 72H |
| 南美 | ≥ 6 条路线，覆盖 24H / 48H / 72H |
| 非洲 | ≥ 6 条路线，覆盖 24H / 48H / 72H |

### 3.4 服务层集成

在 `itinerary_service.py` 的 `generate()` 方法中，将预置查询插入为 step 0：
- 预置命中时记录幂等键（避免重复点击重处理）
- 日志记录 `[request_id] Preset hit for dest=XX hours=YY`
- 若请求体携带 `continent` / `sub_region`，则优先在对应区域内匹配

### 3.5 前端区域开关与默认定位

前端生成页新增两层筛选：
- **大洲开关**：6 大洲切换
- **重点子区域**：东亚、东南亚、英国、欧洲大陆、拉丁美洲、北非、撒哈拉以南非洲

默认大洲推断优先级：
1. GPS 经纬度 → 推断所在大洲
2. 用户拒绝授权或定位失败 → 时区推断
3. 仍无法识别 → 回退 `Asia`

## 4. 配置项变更

本次无新增后端环境变量。前端新增 [`expo-location`](../../frontend/package.json) 依赖以支持 GPS 推断；预置路线数据仍硬编码在 [`preset_routes.py`](../../backend/app/services/preset_routes.py)。

## 5. 风险与回滚

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 预置路线数据过时 | 价格/景点信息可能不准确 | 定期人工更新预置数据；标注"仅供参考" |
| 新目的地无预置覆盖 | 仍然走 LLM 路径（10-60s） | 前端加载文案提示"AI 正在专属定制中" |
| 预置匹配过于宽泛 | 用户可能得到不太匹配的路线 | 评分阈值 5.0 + 50% 容忍范围限制 |

回滚方案：在 `itinerary_service.py` 中注释掉 step 0 预置查询块即可完全回退到纯 LLM 模式。
