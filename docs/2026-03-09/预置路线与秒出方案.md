## 1. 功能目标

在用户点击"开始规划"时，优先匹配内置预置路线（0ms 响应），仅在无匹配时回退至 LLM 生成（10-60s）。
核心目标：消除 80%+ 热门目的地查询的长等待，降低用户流失率。

## 3. 关键实现

### 3.1 预置路线匹配算法

```
preset lookup (0ms) → idempotency → cache (1ms) → LLM (10-60s)
```

匹配逻辑（`find_preset()`）：
1. **目的地模糊匹配**：支持中英文别名（如 `okinawa` → `冲绳`、`saigon` → `胡志明市`），通过 `_DEST_ALIASES` + `_normalize_dest()` 实现
2. **时长容忍**：±50%（如 48H 预置可匹配 24H-72H 查询）
3. **预算容忍**：±50%
4. **标签加分**：tag 交集/并集比率 × 权重，非必要条件
5. **评分阈值**：score < 5.0 视为不匹配

## 5. 风险与回滚

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 预置路线数据过时 | 价格/景点信息可能不准确 | 定期人工更新预置数据；标注"仅供参考" |
| 新目的地无预置覆盖 | 仍然走 LLM 路径（10-60s） | 前端加载文案提示"AI 正在专属定制中" |
| 预置匹配过于宽泛 | 用户可能得到不太匹配的路线 | 评分阈值 5.0 + 50% 容忍范围限制 |

回滚方案：在 `itinerary_service.py` 中注释掉 step 0 预置查询块即可完全回退到纯 LLM 模式。


新增 11 条测试：
- `test_preset_exact_match_okinawa` — 精确匹配冲绳 48H
- `test_preset_exact_match_bangkok` — 精确匹配曼谷 24H

- `test_preset_alias_saigon` — saigon → 胡志明市
