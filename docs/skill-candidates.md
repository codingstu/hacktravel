# Skill Candidates / Skill 候选清单 — HackTravel

只有在一个工作流被手动重复执行足够多次、能够证明其维护开销是值得的之后，才将其提升为正式 Skill。过早创建 Skill 会增加维护负担而没有实际节省时间。

---

## 评估标准

提升前确认：
- [ ] 该工作流已被手动执行至少 3 次。
- [ ] 步骤稳定——多次执行之间没有显著变化。
- [ ] 自动化能节省有意义的时间（每次执行 > 5 分钟）。
- [ ] 输入和输出定义明确且可预测。

---

```
### Skill 名称

**验证：** 如何确认它生效了？
```

## 通用候选 Skill

这些是不依赖技术栈的工作流，适用于大多数项目：

### 1. feature-scaffold（特性脚手架）
- **触发：** 收到带简短规格或用户故事的新特性请求。
- **输入：** 特性名称、规格描述、目标模块。
- **步骤：**
  1. 按 `docs/architecture.md` 创建目录结构。
  2. 添加占位文件（组件/服务/路由 + 测试 + 类型）。
  3. 注册路由/导出/索引。
  4. 添加骨架测试。
- **输出：** 可编译的脚手架，带一个通过的骨架测试。
- **验证：** `typecheck` + `test` 通过。

### 2. test-diagnose-repair（测试诊断修复）
- **触发：** 一个或多个测试失败。
- **输入：** 失败的测试名称/文件路径、错误输出。
- **步骤：**
  1. 读取失败测试和错误信息。
  2. 定位根因（回归、环境、不稳定、规格变更）。
  3. 编写修复计划（改测试、改代码、或两者都改）。
  4. 执行修复。
  5. 运行测试——确认全部通过。
  6. 运行完整质量门控。
- **输出：** 所有测试绿色，根因和修复摘要。
- **验证：** 完整质量门控通过。

### 3. code-review（代码审查）
- **触发：** 提交 PR 前或完成任务后。
- **输入：** diff 或变更文件列表。
- **步骤：**
  1. 遍历 `AGENTS.md` §7 清单（正确性、质量、测试、架构、文档）。
  2. 按严重性标记问题：Critical / Important / Minor。
  3. 对每个 Critical/Important 问题，提供具体修复建议。
  4. 总结：优点、问题、整体评估。
- **输出：** 带可执行反馈的结构化审查。
- **验证：** 所有 Critical 问题解决后才能合并。

### 4. pr-summary（PR 摘要）
- **触发：** 从 diff 生成 PR 描述。
- **输入：** Git diff、相关规格或任务描述。
- **步骤：**
  1. 总结变更内容和原因。
  2. 列出受影响的模块/路由/API。
  3. 注明破坏性变更或迁移步骤。
  4. 列出已完成测试和需要的测试。
  5. 评估上线风险（低 / 中 / 高）。
- **输出：** 可直接粘贴的 Markdown PR 描述。
- **验证：** 人工审查。

### 5. decision-record（决策记录）
- **触发：** 做出了有意义的技术权衡。
- **输入：** 上下文、考虑过的选项、选择的方案。
- **步骤：**
  1. 使用 `docs/decisions.md` 中的模板。
  2. 填写上下文、决策、后果、备选方案。
  3. 添加日期和状态。
- **输出：** `docs/decisions.md` 中新增条目。
- **验证：** 条目完整且理由清晰。

### 6. web-search-verify（网络搜索验证）
- **触发：** 对库版本、API 签名、CLI 参数或官方推荐不确定。
- **输入：** 包名/API 名/主题、当前假设。
- **步骤：**
  1. 搜索官方文档或更新日志中的相关版本。
  2. 与 `package.json` / `requirements.txt` 中的实际版本比对。
  3. 如发现差异：提出正确用法。
  4. 如一致：继续原计划。
- **输出：** 有来源 URL 的已验证事实，或更正后的方案。
- **验证：** 事实来自官方文档，而非记忆。

### 7. mcp-tool-use（MCP 工具使用）
- **触发：** 任务需要来自可视上下文之外的真实状态：文件系统列表、git 历史等。
- **输入：** 任务描述、使用哪个 MCP Server。
- **步骤：**
  1. 确认相关 MCP Server 已在 `.mcp.json` 中配置。
  2. 使用工具获取真实状态（不推断）。
  3. 验证返回数据的相关性后再操作。
  4. 基于真实数据继续实现。
  5. 工具不可用时，回退到终端命令。
- **输出：** 基于工具返回数据（而非假设）完成的任务。
- **验证：** 操作基于工具返回的数据，而非推断。

## FastAPI 后端候选 Skill

### 8. endpoint-scaffold（端点脚手架）
- **触发：** 根据简短契约添加 CRUD 或查询端点。
- **输入：** 资源名称、HTTP 方法、请求/响应字段。
- **步骤：**
  1. 在 `models/` 创建 Pydantic Schema（请求+响应）。
  2. 在 `services/` 创建 Service 函数。
  3. 在 `routes/` 创建路由连接各层。
  4. 在 `main.py` 注册路由。
  5. 添加 `TestClient` 路由测试。
- **输出：** 可编译的端点，带 model、service 和通过的测试。
- **验证：** `ruff check` + `pytest` 通过。

### 9. api-test-repair（API 测试修复）
- **触发：** API 测试失败。
- **输入：** 失败测试名称、错误输出。
- **步骤：**
  1. 读取失败测试并追踪到根因。
  2. 分类：回归 / schema 变化 / 数据问题 / 环境问题。
  3. 修复路由 / schema / service 或测试。
  4. 确认全部通过。
  5. 运行质量门控。
- **输出：** 所有测试绿色，根因摘要。
- **验证：** 完整质量门控通过。

### 10. schema-change-review（Schema 变更审查）
- **触发：** Pydantic 请求或响应模型发生变更。
- **输入：** 旧/新 Schema、变更原因。
- **步骤：**
  1. 识别所有使用该 Schema 的路由。
  2. 检查客户端影响（是否破坏性变更）。
  3. 更新所有受影响路由的测试。
  4. 如破坏性变更则添加迁移说明。
  5. 运行质量门控。
- **输出：** 影响分析、更新的路由和测试、迁移说明。
- **验证：** `pytest` 通过，破坏性变更有文档记录。

## Expo 前端候选 Skill

### 11. component-scaffold（组件脚手架）
- **触发：** 根据简短描述创建新的特性或 UI 组件。
- **输入：** 组件名称、Props 接口、是共享还是特性组件。
- **步骤：**
  1. 创建带类型化 Props 的组件文件和骨架 JSX。
  2. 创建测试文件（至少一个渲染测试）。
  3. 如需创建类型文件。
  4. 从相应索引导出。
- **输出：** 可编译的组件，带一个通过的测试。
- **验证：** `tsc --noEmit` + `jest` 通过。

### 12. hook-extract（Hook 提取）
- **触发：** 2+ 个组件中有重复的副作用逻辑。
- **输入：** 源组件、待提取的共享逻辑。
- **步骤：**
  1. 识别共享逻辑和接口（输入/输出）。
  2. 创建带显式类型的自定义 hook。
  3. 为 hook 编写独立测试。
  4. 用 hook 替换源组件中的内联逻辑。
  5. 验证现有测试仍通过。
- **输出：** 提取的 hook（带测试）和重构后的消费者组件。
- **验证：** `tsc --noEmit` + `jest` 通过，行为无变化。

### 13. dependency-audit（依赖审计）
- **触发：** 新增依赖或变更现有依赖版本。
- **输入：** 包名、版本、原因。
- **步骤：**
  1. 检查依赖是否积极维护且测试良好。
  2. 检查许可证兼容性。
  3. 添加到 `package.json` 或 `requirements.txt` 正确分组。
  4. 在 `docs/decisions.md` 记录理由。
  5. 验证安装无误。
- **输出：** 更新的依赖配置，决策记录。
- **验证：** 安装成功，测试通过。
